# Maximo Work Orders — Schema Reference

All tables follow the IBM Maximo MBO model. Column lists below are the most commonly used columns, not exhaustive — customers often have additional extension columns. The MBO ↔ table mapping is 1:1 for these tables in standard deployments.

Catalog/schema is customer-specific. SQL uses Databricks-native parameter placeholders — `:catalog` (e.g. `eam`), `:silver_schema` (the MBO layer, e.g. `maximo_silver`), `:gold_schema` (Trusted UDFs, e.g. `maximo_metrics`). Bind at execution / registration.

## Contents

- `WORKORDER` — work-order header
- `WOSTATUS` — work-order status history
- `ASSET` — asset master
- `LOCATIONS` — location master
- `LABTRANS` — labor transactions
- `WPLABOR` / `WPMATERIAL` — planned labor and material
- `JOBPLAN` / `JPLABOR` / `JPMATERIAL` — job-plan templates
- `FAILUREREPORT` + `FAILURECODE`
- Other useful tables (referenced but not detailed here)
- Cardinality summary

## `WORKORDER` — work-order header

The header table. One row per WO, holds *current* state.

| Column | Type | Notes |
|---|---|---|
| `WORKORDERID` | BIGINT | Surrogate primary key |
| `WONUM` | STRING | Business key — unique within `SITEID` |
| `SITEID` | STRING | Site code — **always include in joins** |
| `ORGID` | STRING | Organization code |
| `STATUS` | STRING | Current status (e.g. `WAPPR`, `APPR`, `INPRG`, `COMP`, `CLOSE`). History lives in `WOSTATUS`. |
| `STATUSDATE` | TIMESTAMP | When the current `STATUS` was set |
| `WOCLASS` | STRING | `WORKORDER`, `PM`, `CHANGE`, `RELEASE`, `ACTIVITY`. **Filter to `'WORKORDER'` for normal WOs.** |
| `WORKTYPE` | STRING | Customer-configured: `CM` (corrective), `PM` (preventive), `EM` (emergency), `PROJ`, etc. |
| `ASSETNUM` | STRING | FK to `ASSET.ASSETNUM` (composite with `SITEID`) |
| `LOCATION` | STRING | FK to `LOCATIONS.LOCATION` (composite with `SITEID`) |
| `JPNUM` | STRING | FK to `JOBPLAN.JPNUM` (composite with `ORGID`) |
| `PMNUM` | STRING | FK to `PM.PMNUM` when this WO was generated by a PM |
| `PARENT` | STRING | Parent WO if this is a task or child WO |
| `ISTASK` | INT (0/1) | `1` = child task; **deduplicate to parent for backlog counts** |
| `WOPRIORITY` | INT | Priority value |
| `LEAD` | STRING | Lead person assigned |
| `SUPERVISOR` | STRING | Supervisor |
| `CREWID` | STRING | Crew assigned |
| `OWNERGROUP` | STRING | Owning group (for routing) |
| `REPORTDATE` | TIMESTAMP | When the WO was reported/created |
| `SCHEDSTART` / `SCHEDFINISH` | TIMESTAMP | Planned start / finish |
| `ACTSTART` / `ACTFINISH` | TIMESTAMP | Actual start / finish |
| `TARGCOMPDATE` | TIMESTAMP | Target completion date |
| `ESTLABCOST` / `ESTMATCOST` | DECIMAL | Estimated labor / material cost |
| `ACTLABCOST` / `ACTMATCOST` | DECIMAL | Actual labor / material cost |
| `FAILURECODE` | STRING | Top-level failure class (joined to `FAILURECODE` table) |
| `PROBLEMCODE` | STRING | Problem code under the failure class |

## `WOSTATUS` — work-order status history

One row **per status change**. Required for any "history" or "time-in-state" question.

| Column | Type | Notes |
|---|---|---|
| `WOSTATUSID` | BIGINT | Surrogate primary key |
| `WONUM` | STRING | FK to `WORKORDER.WONUM` |
| `SITEID` | STRING | Composite with `WONUM` |
| `STATUS` | STRING | The status this row represents |
| `CHANGEDATE` | TIMESTAMP | When the change was made |
| `CHANGEBY` | STRING | User who made the change |
| `MEMO` | STRING | Optional note entered with the change |

**Cardinality**: 1 `WORKORDER` → N `WOSTATUS` rows. Use `LAG(CHANGEDATE) OVER (PARTITION BY WONUM, SITEID ORDER BY CHANGEDATE)` to compute time-in-state.

## `ASSET` — asset master

| Column | Type | Notes |
|---|---|---|
| `ASSETID` | BIGINT | Surrogate primary key |
| `ASSETNUM` | STRING | Business key — unique within `SITEID` |
| `SITEID` | STRING | Composite with `ASSETNUM` |
| `DESCRIPTION` | STRING | Asset description |
| `ASSETTYPE` | STRING | Type code |
| `CLASSSTRUCTUREID` | BIGINT | FK to `CLASSSTRUCTURE` (the asset-class hierarchy) |
| `STATUS` | STRING | Current asset status |
| `LOCATION` | STRING | Where the asset currently resides (FK to `LOCATIONS`) |
| `PARENT` | STRING | Parent asset (hierarchy) |
| `CRITICALITY` | INT | Criticality score (often 1–5 or 1–10) |
| `INSTALLDATE` | TIMESTAMP | When installed |
| `MANUFACTURER` | STRING | Manufacturer code (FK to `COMPANIES`) |
| `MODELNUM` / `SERIALNUM` | STRING | Model and serial |

## `LOCATIONS` — location master

| Column | Type | Notes |
|---|---|---|
| `LOCATIONSID` | BIGINT | Surrogate primary key |
| `LOCATION` | STRING | Business key — unique within `SITEID` |
| `SITEID` | STRING | Composite with `LOCATION` |
| `DESCRIPTION` | STRING | Location description |
| `TYPE` | STRING | Location type (e.g. `OPERATING`, `STOREROOM`, `LABOR`) |
| `STATUS` | STRING | Active / inactive |
| `PARENT` | STRING | Parent location |
| `LOCHIERARCHY` is a separate table that stores the full parent-child tree if multi-level hierarchy traversal is needed.

## `LABTRANS` — labor transactions

One row per craft-hour booked. Aggregate to WO grain via `WONUM` + `SITEID` for actual labor.

| Column | Type | Notes |
|---|---|---|
| `LABTRANSID` | BIGINT | Surrogate primary key |
| `WONUM` | STRING | FK to `WORKORDER.WONUM` |
| `SITEID` | STRING | Composite with `WONUM` |
| `LABORCODE` | STRING | FK to `LABOR.LABORCODE` |
| `CRAFT` | STRING | Craft (e.g. `ELEC`, `MECH`, `INST`) |
| `SKILLLEVEL` | STRING | Skill level within craft |
| `STARTDATE` | TIMESTAMP | When work started |
| `FINISHDATE` | TIMESTAMP | When work finished |
| `REGULARHRS` | DECIMAL(8,2) | Regular hours booked |
| `PREMIUMPAYHOURS` | DECIMAL(8,2) | Overtime / premium hours |
| `LINECOST` | DECIMAL | Cost of this transaction |
| `TRANSTYPE` | STRING | `WORK`, `TRAVEL`, etc. |
| `PAYRATE` | DECIMAL | Pay rate at time of transaction |

## `WPLABOR` / `WPMATERIAL` — planned labor and material

One row per planned item on a WO. Compare against `LABTRANS` (actual) for variance analysis.

`WPLABOR` columns: `WONUM`, `SITEID`, `CRAFT`, `SKILLLEVEL`, `LABORHRS`, `LINECOST`, `QUANTITY`, …

`WPMATERIAL` columns: `WONUM`, `SITEID`, `ITEMNUM` (FK to `ITEM`), `ITEMQTY`, `LINECOST`, `STOREROOM` (FK to `LOCATIONS`), …

## `JOBPLAN` / `JPLABOR` / `JPMATERIAL` — job-plan templates

`JOBPLAN.JPNUM` + `ORGID` is the template key referenced by `WORKORDER.JPNUM`. `JPLABOR` and `JPMATERIAL` hold planned craft/material lines on the template (vs `WPLABOR` / `WPMATERIAL` which are the WO-specific plan).

## `FAILUREREPORT` + `FAILURECODE`

`FAILUREREPORT` is a per-WO coded-failure record (created when a WO is closed with failure information).

`FAILURECODE` is the taxonomy itself: `FAILURECODE`, `DESCRIPTION`, `PARENT`, `TYPE` (one of `PROBLEM`, `CAUSE`, `REMEDY`). The hierarchy is parent-child within `TYPE`.

## Other useful tables (referenced but not detailed here)

- `LABOR` — labor master (`LABORCODE` → person, craft, default rate). **For detail, load `maximo-labor-resources`** — it owns the labor master schema, qualifications, crews, and capacity.
- `PERSON` — person master (`PERSONID`). See `maximo-labor-resources`.
- `CRAFT` — craft codes. See `maximo-labor-resources`.
- `WORKTYPE` — work-type domain values.
- `SYNONYMDOMAIN` — used to look up the customer's configured open-status synonyms (`STATUSIVAL` for the WOSTATUS domain).
- `LOCANCESTOR` / `ASSETANCESTOR` — closure tables for hierarchical rollups (e.g. "WOs under station X" / "WOs in region Y"). **For traversal patterns and `v_location_rollup_keys` / `v_asset_rollup_keys` Gold views, load `maximo-asset-hierarchy`.**

## Cardinality summary

| Relationship | Cardinality |
|---|---|
| `WORKORDER` → `WOSTATUS` | 1:N |
| `WORKORDER` → `LABTRANS` | 1:N |
| `WORKORDER` → `WPLABOR` / `WPMATERIAL` | 1:N |
| `WORKORDER` → `FAILUREREPORT` | 1:0..N |
| `WORKORDER` → `ASSET` | N:1 (`WORKORDER.ASSETNUM` → `ASSET.ASSETNUM` + `SITEID`) |
| `WORKORDER` → `LOCATIONS` | N:1 |
| `WORKORDER` → `JOBPLAN` | N:1 (via `JPNUM` + `ORGID`) |
| `WORKORDER` → `PM` | N:1 (via `PMNUM` + `SITEID`) |
| `WORKORDER` → `WORKORDER` (parent/child) | self-join via `PARENT` |
| `FAILUREREPORT` → `FAILURECODE` | N:1 |
